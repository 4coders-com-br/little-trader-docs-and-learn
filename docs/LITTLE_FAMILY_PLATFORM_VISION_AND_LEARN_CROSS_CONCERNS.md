# Little Family Platform Vision + Learn Cross-Concerns

## Vision

Little Trader should not remain an isolated product.

It should become the first strong expression of a broader family of systems built from the same philosophy:

- data-driven
- immutable where it matters
- local-first where useful
- AI-assisted and AI-operated
- REPL-first
- rules-engine friendly
- MCP-enabled
- strong DevOps
- rich UI
- deeply learnable by humans

This document expands the idea into a family:

- **Little Trader** — trading, research, execution, strategy, hedge-fund thinking
- **Little Business** — commerce, services, industry, operations, CRM, CMS, sales, social channels
- **Little Store** — ecommerce-first business operating system
- **Little Sys** — shared systems substrate and domain-agnostic platform foundation

And pulls all shared concerns back into the **Learn** surface.

## Shared Philosophy Across the Family

All family members should inherit the same deep beliefs.

### 1. Data-Driven
- operational state should be inspectable
- decisions should be grounded in visible data
- dashboards are not decoration; they are instruments

### 2. Immutable Where It Matters
- auditability
- replayability
- decision history
- event traceability

### 3. Local-First Where Useful
- resilient client workflows
- optimistic interaction
- offline or degraded survivability
- fast rich UI

### 4. AI-Assisted and AI-Operated
- AI should explain, guide, summarize, compare, simulate
- AI may operate bounded tasks with explicit guardrails
- AI must not become invisible authority

### 5. REPL-First
- inspect before guessing
- code execution available where safe
- runtime state should be interrogable
- even the client should have a disciplined execution surface

### 6. Rules Engine as a First-Class Primitive
- policies
- workflows
- risk logic
- validation
- eligibility
- automations
- decision envelopes

### 7. MCP-Native
- bridge the product with tools, models, dev workflows, and operator workflows
- allow structured action surfaces instead of only chat

### 8. Strong DevOps
- parity between local and cloud
- reproducible environments
- rich observability
- resilient automation
- staged rollouts and rollback discipline

### 9. Rich UI
- not sterile CRUD
- guided attention
- visual explanation
- interactive learning
- stateful operator experience

### 10. Learnability as a Product Requirement
- every product should teach the user how to think inside the domain
- Learn is not optional documentation; it is part of the product architecture

## Shared Architecture Rings

All Little-* platforms can be seen as concentric rings.

### Ring 1 — Domain
The unique language of the domain.

Examples:
- Trader: positions, volatility, risk, signal
- Business: lead, deal, margin, ticket, supplier
- Store: product, order, campaign, customer, cart
- Sys: event, identity, policy, execution, environment

### Ring 2 — Application Workflows
The workflows humans perform.

Examples:
- open trade
- qualify lead
- fulfill order
- troubleshoot system drift

### Ring 3 — Decision Layer
Where rules, AI, and operators interact.

Examples:
- should this trade be taken?
- should this lead be escalated?
- should this order be flagged?
- should this deployment continue?

### Ring 4 — Runtime / Data Layer
Where state, history, caches, projections, and sync live.

### Ring 5 — Learn / Mentor Layer
Where humans are taught how to understand and operate the system.

### Ring 6 — Platform / Ops Layer
Where DevOps, observability, security, environments, and automation live.

## Cross-Cutting Concerns Shared by All Little-* Products

- what actions are safe?
- what actions require confirmation?

### Execution Engine
- client-side safe execution

- task orchestration
- scheduled automations

### Observability
- health
- metrics
- logs
- traces
- domain KPIs

### Learnability
- guided tours
- snippets
- wizards
- context-sensitive teaching
- mentor overlays

### Personal Knowledge / Brain Layer
- notes
- decisions
